# Web Package Instructions

When working in `packages/web/`:

## Data Fetching — TanStack Query
All server data is fetched via TanStack Query. Never use raw `fetch` or `axios` directly in components.

```typescript
import { useQuery, useMutation, useQueryClient } from '@tanstack/react-query';
import { api } from '../lib/api';

// Read
const { data, isLoading } = useQuery({
  queryKey: ['entities', filters],
  queryFn: () => api.get('/entities', { params: filters }).then(r => r.data.data),
});

// Write
const queryClient = useQueryClient();
const mutation = useMutation({
  mutationFn: (data) => api.post('/entities', data).then(r => r.data.data),
  onSuccess: () => queryClient.invalidateQueries({ queryKey: ['entities'] }),
});
```

## State Management
- **Server state**: TanStack Query only — queries auto-invalidate on WebSocket events
- **Auth state**: Zustand store at `src/stores/auth.ts` — the only Zustand store
- **UI state**: Local React state (`useState`, `useReducer`) — never put UI state in Zustand or TanStack Query

## Real-Time Updates
WebSocket messages from the server trigger automatic query invalidation in `src/lib/useWebSocket.ts`. When adding new entity types:
1. Add the message type to the WebSocket handler
2. Map it to the correct TanStack Query key for invalidation

## File Organization
- Pages: `src/pages/XxxPage.tsx` — one component per route
- Components: `src/components/` — reusable across pages
- Hooks: `src/hooks/` — custom hooks for shared logic
- API client: `src/lib/api.ts` — configured axios instance with auth headers

## Styling
- TailwindCSS utility classes only — no CSS modules, no styled-components
- Use display name maps from `@erp/shared` for labels:
  ```typescript
  import { STATUS_DISPLAY_NAMES, STATION_DISPLAY_NAMES } from '@erp/shared';
  ```
- Never hardcode enum display values in JSX

## Types
- Import all types from `@erp/shared` — never define API types locally
- Prisma types are available through shared package re-exports
