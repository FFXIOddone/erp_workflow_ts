# Shared Package Instructions

When working in `packages/shared/`:

## This Package is the Source of Truth
Every type, enum, Zod schema, and constant used across server and frontend lives here. Never define types inline in server routes or frontend components.

## File Order for New Features
Always add in this sequence:
1. **`src/enums.ts`** — Add new enum values. Must match Prisma enums exactly.
2. **`src/types.ts`** — Add interfaces/types that reference the enums.
3. **`src/schemas.ts`** — Add Zod schemas for API validation (e.g., `CreateXxxSchema`, `UpdateXxxSchema`).
4. **`src/constants.ts`** — Add display name maps (e.g., `XXX_DISPLAY_NAMES`).
5. **`src/index.ts`** — Export everything new.

## Enum Rules
- Enum values must be `UPPER_SNAKE_CASE`
- Every Prisma enum must have an identical TypeScript enum here
- When adding a new enum value, update both `enums.ts` AND `schema.prisma`

## Zod Schema Rules
- Naming: `Create___Schema`, `Update___Schema`, `___FilterSchema`
- Use `.optional()` for Update schemas (partial updates)
- Use `z.nativeEnum(MyEnum)` to validate enum fields
- Export the inferred types: `export type CreateXxx = z.infer<typeof CreateXxxSchema>`

## Constants Rules
- Display name maps: `Record<EnumType, string>` mapping enum values to human-readable labels
- Example: `export const STATUS_DISPLAY_NAMES: Record<OrderStatus, string> = { ... }`
- Used by frontend for labels — never hardcode display strings in components

## Export Rules
- `src/index.ts` re-exports everything — this is what `@erp/shared` resolves to
- Always add new exports here when creating new types/schemas/enums
- Group exports by domain area with comments
